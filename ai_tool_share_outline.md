# AI 工具分享大纲（算法工程师向｜Trae 常用功能）

## 0. 分享定位与目标（3分钟）
- **受众画像**：算法工程师、ML 工程师、数据科学同学。
- **分享目标**：
  1. 认识 Trae 在“需求拆解、编码实现、调试验证、交付沉淀”中的关键能力。
  2. 能把 AI 工具嵌入日常算法开发链路（数据处理→模型开发→评估上线）。
  3. 掌握“高质量提问模板”，提升输出稳定性与可复用性。

## 2. Trae IDE 重点功能汇总（15分钟）
> 参考链接：<https://docs.trae.cn/ide>

### 2.1 Solo 模式（个人高效开发模式）
- 核心理解：面向个人任务闭环，强调“你提目标，AI帮你推进实现”。
- 适合算法工程师：
  - 快速产出 baseline、实验脚本、可复现实验说明。
  - 在个人探索阶段降低上下文切换成本。

### 2.2 Plan 与 Spec（先规划再编码）
- **Plan**：先拆解任务步骤，明确优先级与依赖。
- **Spec**：把需求固化为可执行规格（输入/输出/边界/验收标准）。
- 算法场景价值：
  - 先定义评估指标（AUC/F1/Recall）再写训练代码。
  - 先写“实验设计 spec”再批量跑实验，减少无效试错。

### 2.3 MCP 与 Skill（能力扩展与复用）
- **MCP**：把外部工具/数据源接入到智能体工作流。
- **Skill**：将高频任务方法论沉淀为可复用能力模板。
- 算法场景价值：
  - 把“数据巡检、特征检查、实验报告”做成团队可复用 Skill。
  - 结合内部平台/知识库，缩短从问题到答案路径。

### 2.4 上下文压缩（Context Compression）
- 核心理解：在超长对话或大仓库中压缩关键上下文，保留任务有效信息。
- 算法场景价值：
  - 多轮调参时保留关键结论（模型版本、参数、指标变化）。
  - 减少 token 浪费，提升对话连续性和回复稳定性。

### 2.5 沙箱（Sandbox）
- 核心理解：在受控环境中执行命令、验证脚本，降低对主环境的风险。
- 算法场景价值：
  - 先在沙箱验证数据清洗脚本和依赖安装。
  - 降低“改坏训练环境/误删文件”的风险。

### 2.6 智能体（Agent）
- 核心理解：把复杂任务交给 AI 作为“可执行代理”分步骤推进。
- 算法场景价值：
  - 自动执行“读代码→改代码→跑检查→总结结果”链路。
  - 适合排障、重构、文档整理等多步骤任务。

## 3. 结合算法开发的实际案例（12分钟）

### 案例1：基础流程（4分钟）
**场景**：快速搭建一个用户流失预测 baseline。
- AI 协助内容：
  1. 用 Plan 拆解任务（数据、训练、评估、交付）。
  2. 用 Spec 固化输入输出、指标与验收标准。
  3. 在 Solo 模式中生成可运行 pipeline（预处理 + 训练 + 评估）。
- 参考链接：<https://qianli-drive.feishu.cn/wiki/JAENwwvKEijha9kZztTcnlcRngd>
- 产出：可运行 baseline + 初版实验报告。

### 案例2：异常排查（4分钟）
**场景**：深度模型训练第3轮出现 nan。
- AI 协助内容：
  1. 智能体分步执行日志分析与可疑点定位。
  2. 借助上下文压缩保留关键排障线索。
  3. 在沙箱先验证修复方案（梯度裁剪、`eps`、数据范围检查）。
- 产出：可复现问题 + 修复验证闭环。

### 案例3：新功能添加（4分钟）
**场景**：在现有算法项目中新增一个“实验配置自动生成+任务追踪”功能。
- AI 协助内容：
  1. 用 Spec 先定义新功能边界（输入配置、输出脚本、失败重试规则）。
  2. 结合个人规则与工程规则，统一代码风格、目录约定与提交流程。
  3. 通过 MCP 对接 GitLab 的 projectmanagement 能力，自动同步任务状态与变更记录。
- 产出：新功能开发闭环 + 可追踪的项目协同记录。

## 4. 高效使用 Trae 的方法论（3分钟）

### 4.1 Prompt 结构（推荐模板）
- 背景：任务与业务目标。
- 上下文：相关文件、函数、错误日志。
- 约束：技术栈、性能目标、代码风格。
- 输出：期望格式（代码 / diff / 表格 / 结论）。

### 4.2 “先小后大”迭代策略
- 先让 AI 给最小可运行版本（MVP）。
- 再补边界条件、性能优化、可观测性。
- 最后补单测与文档。

### 4.3 质量保障清单
- 关键路径必须人工 review。
- 单测覆盖核心逻辑。
- 指标回归（离线+线上）通过后再合并。

## 6. 互动答疑参考回答（含简单例子）

### Q1：Solo、Agent、Plan/Spec 在不同项目阶段怎么搭配？
- 建议搭配：
  - 需求澄清期：先用 Plan/Spec 对齐范围与验收。
  - 实现推进期：用 Solo 快速产出 MVP。
  - 联调收敛期：用 Agent 执行“改动-验证-总结”闭环。
- 例子：做推荐模型重排功能时，先写 Spec（输入特征、输出排序、延迟阈值），再用 Solo 生成 baseline，最后让 Agent 跑回归脚本并汇总指标。

### Q2：如何把 MCP + Skill 落地为团队资产？
- 建议做法：
  1. 先挑高频任务（数据巡检、实验报告、发布检查）。
  2. MCP 接内部系统（GitLab、实验平台、知识库）。
  3. 把提示词+步骤固化成 Skill，并沉淀版本。
- 例子：每次训练后自动调用 MCP 读取实验指标，再由 Skill 生成统一周报模板，团队直接复用。

### Q3：上下文压缩对长链路调参与故障排查的最佳实践是什么？
- 建议做法：
  - 每轮只保留“参数改动→指标变化→结论”。
  - 固定输出三段：现象、可能原因、下一步验证。
  - 到关键节点做一次压缩快照，避免信息漂移。
- 例子：调学习率 5 轮后，把每轮结果压成 6 行摘要，下一轮直接基于摘要继续，而不是回翻全部日志。

### Q4：沙箱验证应该卡在哪些关键环节？
- 建议卡点：
  1. 新依赖安装前。
  2. 高风险脚本执行前（批量改数据、删文件、迁移脚本）。
  3. 合并前回归检查。
- 例子：清洗脚本要改 20 万条样本，先在沙箱抽样跑 1% 数据，确认无误再跑全量。

### Q5：算法团队最该优先落地哪些 AI 能力？
- 优先级建议：
  1. Plan/Spec（减少返工）。
  2. Agent + 沙箱（提升排障效率且可控）。
  3. MCP + Skill（团队复用与规模化）。
- 例子：先统一“实验 Spec 模板”，两周内就能明显减少“指标口径不一致”的沟通成本。

### Q6：如何平衡“开发效率”与“模型可靠性”？
- 原则：AI 负责加速，人工负责兜底。
- 落地动作：
  - 把“必须人工 review”写进工程规则（核心损失函数、线上特征逻辑、阈值策略）。
  - 把可靠性检查自动化（单测、回归、数据漂移检查）。
- 例子：AI 可以快速改特征工程代码，但上线前必须人工确认离线/在线特征一致性。

### Q7：哪些任务适合全交给 AI，哪些必须人工主导？
- 适合 AI 主导：模板化与重复性任务（脚手架、日志汇总、文档整理、批量重构）。
- 必须人工主导：业务目标定义、关键架构决策、上线审批与风险兜底。
- 例子：AI 可自动生成训练脚本注释；但“是否切换线上主模型”必须由负责人拍板。

### Q8：如何在团队内建立 prompt 与脚本资产库？
- 建议结构：
  - prompts/：按场景分类（排障、调参、代码评审）。
  - scripts/：可执行脚本（评估、回归、报告）。
  - examples/：输入输出样例。
  - CHANGELOG：记录版本演进。
- 例子：新同学遇到 nan，直接套用“nan 排障 prompt + 检查脚本”，30 分钟内完成初步定位。

### Q9：从个人提效到团队流程改造，第一步做什么？
- 第一步：选一个“高频、低风险、可量化”的流程做试点。
- 衡量指标：耗时、返工率、缺陷率。
- 例子：先试点“实验报告自动生成”，若每周节省 4 小时，再扩展到数据巡检与发布检查。

### Q10：Solo、Agent、Plan/Spec 与 MCP/Skill 如何组合落地？
- 推荐闭环：
  1. Plan/Spec 定义任务和验收。
  2. Solo 快速实现 MVP。
  3. Agent 执行测试与修复迭代。
  4. MCP 对接外部系统拿数据/写状态。
  5. Skill 固化为团队标准流程。
- 例子：新增“特征质量看板”时，先写 Spec，再用 Solo 生成功能，Agent 跑检查，MCP 同步 GitLab 任务，最后沉淀为 Skill 供后续项目复用。
